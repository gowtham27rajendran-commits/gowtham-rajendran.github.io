Why This Architecture?
  • Why Medallion over single-layer ETL? Separates concern: Bronze = raw audit trail, Silver = clean validated, Gold = business-ready. Each layer independently queryable and reprocessable.
  • Why Delta Lake on Bronze/Silver? ACID transactions prevent partial writes from corrupting downstream. MERGE operation enables idempotent upserts — critical for exactly-once streaming.
  • Why Apache Iceberg on Gold? Partition evolution without full table rewrite. Hidden partitioning. Time-travel with snapshot isolation. Cross-engine (Trino + Spark) interop.
  • Why Airflow for orchestration? Sensor-based dependencies, SLA monitoring, backfill for any date range, and idempotent retries on failure — non-negotiable for production data pipelines.

Where Latency & Bottlenecks Happen
Bottleneck Analysis
  • Small files problem: Streaming creates many tiny Parquet files. Solved with Spark OPTIMIZE + ZORDER scheduled via Airflow after each microbatch.
  • Schema evolution lag: New columns in Bronze don't auto-propagate to Gold. Solved by registering schema changes in Glue Catalog and running schema migration DAG.
  • Shuffle in Silver MERGE: Broadcast join hint used for small dimension tables; large fact tables repartitioned on entity_id before merge.
  • Checkpoint overhead: Structured Streaming checkpoints to S3. High-frequency writes can throttle. Mitigated with checkpoint compaction and S3 bucket lifecycle rules.
Key Implementation
delta_merge_silver.py — ACID upsert
def silver_upsert(spark, bronze_path, silver_path):
    bronze = spark.readStream.format("delta").load(bronze_path)\
        .withColumn("row_hash", sha2(concat_ws("|", *KEY_COLS), 256))\
        .filter("event_ts IS NOT NULL AND user_id IS NOT NULL")

    silver = DeltaTable.forPath(spark, silver_path)

    def merge_batch(batch_df, batch_id):
        silver.alias("t").merge(batch_df.alias("s"),
            "t.row_hash = s.row_hash")\
        .whenNotMatchedInsertAll()\
        .whenMatchedUpdate(
            condition="s.updated_at > t.updated_at",
            set={"t.updated_at":"s.updated_at","t.payload":"s.payload"}
        ).execute()

    bronze.writeStream.foreachBatch(merge_batch)\
        .option("checkpointLocation", f"{silver_path}/_ckpt")\
        .trigger(processingTime="5 minutes").start()

Why This Architecture?
  • Why Schema Registry + Avro? Enforces producer-consumer contracts at the byte level. BACKWARD compatibility means new producers can't break existing consumers — critical for multi-team platforms.
  • Why EOS (transactional producers)? Without transactions, a crash between produce and offset commit causes duplicate processing. EOS guarantees exactly-once even across partitions and consumer groups.
  • Why manual offset commit? Auto-commit marks messages as processed even if processing failed. Manual commit only after successful processing prevents data loss.
  • Why Redis dedup on consumers? Even with EOS producers, consumer restarts can re-process. Redis TTL-based event_id check provides application-level idempotency as a second safety layer.

Why This Architecture?
  • Why dual-store (Redis + S3)? Online inference requires sub-5ms. S3 can't do that. S3 has TB-scale storage for training. Redis has TTL per feature. Two different access patterns → two stores.
  • Why point-in-time joins? The most common silent ML bug. Without PIT, a feature computed at T+1 leaks into a label at T, making the model cheat during training and fail in production.
  • Why semantic versioning on features? Models pin to specific feature versions. A breaking change (type change, logic change) requires a version bump. Consumers declare what they were trained on.
  • Why PSI for drift? PSI measures distribution shift vs training baseline — not just mean shift. PSI > 0.2 = significant distribution change, indicating the model is operating out-of-distribution.

Why This Architecture?
  • Why Base62 not UUID? UUIDs are 36 chars. Base62 encodes a counter to 6–8 chars. Short = better UX, less storage, faster hash lookup. Atomic DB counter prevents collisions without distributed coordination.
  • Why write-through cache? Writes go to both Redis and Postgres atomically. Redis serves all reads. Cache miss → DB fallback → re-warm. This achieves <10ms p99 even without pre-warming.
  • Why Kafka for analytics? Click counting is not in the redirect hot path. Async Kafka event means the redirect returns in <5ms regardless of analytics load.
  • Where latency happens: Cache miss hits Postgres (~5ms). Hot URLs always hit Redis (<1ms). Cold start after deployment until cache warms = only time p99 degrades.

Why This Architecture?
  • Why FAISS IVF not exact search? Exact k-NN is O(n·d). IVF clusters vectors into buckets, searches only nearby clusters. 10–100× faster with <5% recall loss at scale.
  • Why hybrid (dense + sparse)? Dense retrieval misses exact keyword matches ("CVE-2024-1234"). Sparse retrieval misses semantic matches ("fast DB" ≠ "high-performance database"). RRF captures both.
  • Why RRF over score fusion? Dense (cosine 0–1) and sparse (BM25 0–∞) scores are incomparable scales. RRF uses rank positions, which are scale-invariant. More robust than score normalization.
  • Where latency happens: Embedding inference (~8ms on CPU). FAISS IVF search (~3ms). BM25 lookup (~1ms). RRF merge (~0.5ms). Total ~12ms. GPU drops embed to <2ms.

Why This Architecture?

WHY LUA SCRIPTS (not plain Redis)?
  Problem: GET → check → SET has race window
  Thread A reads: 99/100 requests
  Thread B reads: 99/100 requests
  Both write: 100/100 (should be 101!)

  Lua executes ATOMICALLY in Redis server
  GET + check + SET = 1 atomic operation
  Zero race conditions, works across app servers
Algorithm Trade-offs
  • Token bucket: Allows burst up to bucket size. Best for APIs that tolerate short spikes. O(1) space: just (tokens, last_refill) per key.
  • Sliding window log: 100% accurate — no boundary spike. Stores all request timestamps in sorted set. O(n) space — expensive for high-rate APIs.
  • Sliding window counter: O(1) space. ~5% error vs log. Weighted blend of current + previous window counts. Best for high-throughput services.
  • Fixed window: Simplest. Vulnerable to boundary burst (2× rate at window edge). Use only when strict accuracy isn't required.

Why This Architecture?
  • Why Bayesian HPO over grid search? Grid search is O(n^k) combinations. Bayesian TPE builds a probabilistic model of which hyperparameter regions are promising, converging in 3× fewer trials.
  • Why MedianPruner? Stops unpromising trials early based on intermediate F1 scores. Eliminates ~40% of trial compute without losing best configuration.
  • Why SHAP in the pipeline? Model selection shouldn't just maximize F1. SHAP reveals if the model is learning spurious correlations. A model with worse F1 but clean SHAP is often better in production.
  • Why MLflow versioning? Without versioning, you can't reproduce a model from 3 months ago. MLflow logs exact params, metrics, and artifacts — critical for regulatory audit trails.

Why This Architecture?
  • Why deterministic hash for A/B? Random assignment causes the same user to see different variants across sessions. Hash(user_id + experiment_id) is stable — user always gets the same variant. Required for valid statistical inference.
  • Why hybrid (not pure CF)? Pure CF has cold-start problem: new users/items have no interaction history. Content-based fills the gap. Weight (0.7/0.3) is a tunable parameter optimized via offline A/B.
  • Why cache recommendations? ALS inference is O(users × factors). Computing at request time adds 50–200ms. Pre-compute per-user recs hourly, cache in Redis. Sub-1ms serving latency.

Why This Architecture?
  • Why HMAC webhook verification? Without it, anyone can POST to your endpoint and trigger code reviews. GitHub signs every webhook with your secret — verify or reject. Non-negotiable for any GitHub App.
  • Why Shannon entropy for secret detection? Regex catches known patterns (AWS keys, GitHub tokens). High-entropy strings catch unknown formats — random-looking strings are almost always secrets. Combined approach beats either alone.
  • Why stateless design? Lambda/ECS Fargate can scale horizontally. No session state means any instance handles any PR. Idempotent — re-running on same PR SHA produces same review.
  • Where latency comes from: Diff parsing ~100ms. Secret scan ~50ms. LLM API ~5–15s (dominant). GitHub API post ~500ms. Total ~60s for typical PRs.

Why This Architecture?
  • Why 3 pillars together? Metrics show something is wrong. Traces show where. Logs show what. All three are needed for production incident investigation. Single-pillar observability is incomplete.
  • Why W3C trace propagation? Standard headers (traceparent, tracestate) mean any W3C-compliant service in the call chain automatically participates in distributed traces — no custom headers to maintain.
  • Why sustained alerting windows? Single-window alerts fire on brief spikes (deploy noise, GC pause) causing alert fatigue. Requiring N consecutive windows of violation before paging ensures real incidents — no flapping.
  • Why TimescaleDB over raw Prometheus? Prometheus retention is limited. TimescaleDB is Postgres-native time-series with compression — long-term metric storage for capacity planning and SLA reporting.
