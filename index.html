Architecture — Three-Layer Decision System
Transaction Event (POST /score) │ ▼ [ LAYER 1: Velocity Rules ] <1ms Redis HGET/HINCRBY + Lua script (atomic) — Block if: count>15/min OR amount>$5000/5min │ (pass if clean) ▼ [ LAYER 2: ML Scoring ] ~18–45ms Feature fetch from Redis online store XGBoost inference (joblib, no cold start) Score: 0.0–1.0 · Threshold: 0.7 │ ▼ [ LAYER 3: SHAP ] +3–8ms Cached TreeExplainer · top-3 features Required for regulatory compliance │ ▼ APPROVE / REVIEW / BLOCK → Kafka (audit) · PostgreSQL (log)
What I Built
  • Layer decomposition for latency budgeting — cheap rules first (<1ms), ML second (~30ms avg), explainability last; only compute what you need for each decision
  • Class imbalance recovery — default XGBoost gave 12% recall on fraud class. scale_pos_weight=200 + Optuna threshold sweep → 91.4% recall, 96.2% precision
  • SHAP cached at startup — TreeExplainer init was 134ms per request in hot path. Pre-loading at startup → ~5ms per call; p99 stayed under 100ms at 1K RPS
Hardest Problem: Class Imbalance
  • Fraud is <0.5% of transactions. Default XGBoost optimizes for majority class — 99.1% precision but only 12% recall. Completely useless in production.
  • Fix: scale_pos_weight=200 shifts learning toward minority class. Optuna sweeps threshold from 0.3–0.9 to find F1-maximizing cutpoint. Result: recall went from 12% → 91.4%.
WARN Model recall=0.124 on validation set class_1 (fraud) F1: 0.22 — useless ERROR SHAP TreeExplainer: 134ms per request — init in hot path, blocks every call FIX scale_pos_weight=200 + threshold=0.70 explainer cached at startup → ~5ms/call
Why XGBoost over a neural net?
Neural nets need large datasets to outperform tree methods on tabular data — and they're opaque. XGBoost at this scale gives comparable accuracy, 10× faster inference, and plays well with SHAP for compliance. For a regulated fintech use case where a reviewer needs to explain every block decision, SHAP + XGBoost is significantly better than a black-box deep model.

Why Redis for online serving over Postgres?
Postgres with connection pooling gets to ~5–15ms per feature fetch. Redis HGETALL on a pre-computed entity key is sub-millisecond. In a fraud scoring hot path where the entire budget is <100ms, spending 15ms on feature retrieval is not acceptable. Redis also handles the HSET field-level update pattern cleanly for incremental feature updates without full row rewrites.

What I Built
  • Hybrid retrieval — dense handles semantic similarity; BM25 handles exact term matching; RRF merges both. Dense-only Recall@10 was 91%; hybrid got to 94% (+3 points on eval set)
  • FAISS IVFFlat index — nlist=256, nprobe=16; ~10× faster than brute-force at <2% recall loss; hot-reload without downtime
  • Offline eval harness — Recall@10, MRR@10, NDCG@10 computed on held-out query set; every architecture change tracked numerically
Hardest Problem: Exact Code Failures
  • Querying "SKU-4892-B" — dense retrieval returned Recall@10 = 0.0. MiniLM has no semantic signal for product codes; the embedding was noise.
  • BM25 handled it immediately (exact match). Hybrid RRF: Recall@10 = 1.0. This drove the decision to use hybrid rather than optimizing the dense model further.
WARN Query "SKU-4892-B" dense Recall@10=0.0 MiniLM: zero signal for product codes INFO BM25 Recall@10=1.0 (exact match) Hybrid RRF Recall@10=1.0 ✓ Dense-only: 91% · Hybrid: 94% Recall@10
Why BM25 over pure dense retrieval?
Dense models (MiniLM, MPNet) are excellent at semantic similarity but fail on exact token matching — product codes, model numbers, proper nouns. BM25 is the opposite: perfect for exact terms, blind to meaning. Real-world search queries include both. RRF fusion with k=60 doesn't require calibrating scores between systems — it just combines ranks, which generalizes better across query types than score interpolation.

Why Kafka over RabbitMQ?
Kafka
RabbitMQ excels at task queues — routing, priority, complex topologies. Kafka wins for event streaming: durable log, replay, partitioned horizontal scale, consumer group semantics, and exactly-once delivery. When I need ML pipelines consuming the same stream as analytics — Kafka is the only answer.
Why Redis over DB cache?
Redis
Postgres with a caching layer is 10–50ms per query. Redis is sub-millisecond with data structures built for counters, sorted sets, and Lua atomicity. In a fraud detection hot path or rate limiter, that 50ms difference is the entire latency budget.
Why Spark over Flink?
Spark
Flink is better for true streaming with sub-second latency. Spark wins for micro-batch (5–60min windows), massive batch processing, and unified batch+streaming in one framework. For medallion pipelines and feature computation, Spark's ecosystem (Delta Lake, MLlib) is unmatched.
Principles
ENGINEERING
THINKING
01
Design for Failure
Assume every component will break. Build circuit breakers, DLQs, retries with backoff, and fallback paths before they're needed — not after the incident.
02
Measure Everything
Latency percentiles, throughput, error rates, drift. Every architectural decision should be backed by numbers, not intuition. Gut feelings are hypotheses — metrics are answers.
03
Prefer Horizontal Scale
Vertical scaling has a ceiling. Systems designed around partitioning, stateless workers, and shared-nothing architecture scale without rewrites.
04
Push Compute Async
If it doesn't need to happen in the request path, don't put it there. Move expensive work to Kafka consumers, Airflow DAGs, and background workers.
05
Immutable Data Models
Append-only event logs are replayable, debuggable, and auditable. Mutable state is a debt you pay at the worst possible time — during an incident.
06
Own the Tradeoffs
Every tech choice is a tradeoff. Understand what you're giving up, document it, and revisit it when constraints change. There is no universally correct stack.
