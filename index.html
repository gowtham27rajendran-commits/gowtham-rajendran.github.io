Why This Architecture?
  • Why Medallion over single-layer ETL? Bronze = raw audit trail, Silver = validated clean, Gold = business-ready. Each layer independently reprocessable — critical when upstream data quality issues are discovered.

Why This Architecture?
  • Why EOS (transactional producers)? Without transactions, a crash between produce and offset commit causes duplicate processing. EOS guarantees exactly-once even across partitions and consumer groups.
  • Why Redis dedup on consumers? Even with EOS producers, consumer restarts can re-process. Redis TTL-based event_id check provides application-level idempotency as a second safety layer.
  • Why manual offset commit? Auto-commit marks messages as processed even if processing failed. Manual commit only after successful processing prevents silent data loss.

Design Decisions
  • Why Base62 not UUID? UUIDs are 36 chars. Base62 encodes a counter to 6–8 chars. Atomic DB counter prevents collisions without distributed coordination overhead.
  • Why write-through cache? Writes go to both Redis and Postgres atomically. Redis serves all reads. Cache miss → DB fallback → re-warm. Achieves <10ms p99.
  • Why Kafka for analytics? Click counting is not in the redirect hot path. Async Kafka means redirect returns in <5ms regardless of analytics load.

Design Decisions
  • Why hybrid? Dense retrieval misses exact keyword matches ("CVE-2024-1234"). Sparse misses semantic matches ("fast DB" ≠ "high-performance database"). RRF captures both.

  • Why FAISS IVF? Exact k-NN is O(n·d). IVF clusters into buckets, searches only nearby clusters. 10–100× faster with <5% recall loss at scale.

WHY LUA SCRIPTS?
  Problem: GET → check → SET has race window
  Thread A reads 99/100 → Thread B reads 99/100
  Both write 100/100 — should be 101!
  Lua executes atomically in Redis server.
  GET + check + SET = 1 atomic op. No races.
Algorithm Trade-offs
  • Token bucket: Allows burst up to bucket size. Best for APIs that tolerate short spikes. O(1) space per key.
  • Sliding window log: 100% accurate, no boundary spike. Stores all timestamps in sorted set — O(n) space, expensive for high-rate APIs.
  • Sliding window counter: O(1) space, ~5% error vs log. Weighted blend of current + previous window. Best for high-throughput.
